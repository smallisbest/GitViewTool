How gt is built

gt is a single self-contained command-line binary. It never modifies your repository; instead it keeps all of its state in a private directory tree and serves per-view virtual filesystems on top of your existing Git content.

On-disk layout

All gt-managed state lives under ~/.gt/ (overridable via the GT_HOME environment variable):

~/.gt/                          # override with $GT_HOME
├── registry.db                 # path → repo-id index
├── log/
│   └── gt.log                  # rotating log (5 × 1 MB)
└── repos/<12-hex-id>/
    ├── metadata.db             # views, checkouts
    ├── config.toml             # default_cs_branch, default_remote
    ├── views/<view>/
    │   └── overlay/            # writable shadow
    └── mounts/<view>/          # default FUSE mount point

Each repository registered with gt mkvob gets its own subdirectory identified by a 12-hex digest of its canonical absolute path, so renaming a repo directory produces a new gt identity.

View isolation

A view is a named workspace with three pieces of state:

  1. A record in the metadata database: name, kind (Phase 1 is always dynamic), pinned branch, mount path.
  2. An overlay directory — a real directory where edits to checked-out files live.
  3. A mount point — the directory where the FUSE backend exposes the view to the user.

Two views on the same branch are entirely independent. They share unmodified file contents via Git (same blobs), but their overlays, checkout registries, and mount points are separate.

The FUSE layer

When you run gt mount <view>, gt starts a FUSE backend that:

  1. Resolves the view's pinned branch to a commit OID.
  2. Walks the tree of that commit and builds an in-memory index keyed by absolute path within the mount.
  3. Answers POSIX filesystem calls:
    • getattr / readdir — served from the in-memory index; modification times come from the commit's timestamp.
    • open / read — lazy-reads the blob from libgit2 the first time, then caches it in-process.
    • write / truncate — only allowed if the path has a live checkout record; writes go to the overlay file.
    • readlink — returns the blob contents of a symlink entry.

macOS hosts talk to the backend through FUSE-T's NFS-v4 loopback transport, so no kernel extension is required.

Checkout enforcement

gt's checkout registry sits in the per-repo SQLite database and enforces two rules:

  • Reserved exclusivity — a partial unique index on (element_path) WHERE mode = 'reserved' guarantees at most one view may hold a reserved checkout on a given element at any time.
  • Unreserved concurrency — multiple views may hold unreserved checkouts on the same element; first-to-commit wins, and subsequent check-ins are warned about the base having moved.

The FUSE backend consults this registry on every write path. Files without a checkout are surfaced with EROFS at open time, preventing accidental edits.

How check-in works

  1. Read the overlay file's bytes.
  2. Write them as a new Git blob via libgit2.
  3. Clone the current branch-tip tree, replacing the leaf at the element's path with the new blob OID.
  4. Create a commit parented at the branch tip, authored with the user's normal user.name / user.email Git configuration.
  5. Fast-forward the branch reference to the new commit.
  6. Remove the checkout record and delete the overlay file.

The result is indistinguishable from a commit made via plain Git. git log, git push, and continuous integration pipelines all treat it identically.

Coexistence with plain git

gt and plain git can safely share the same working repository:

  • gt never touches .git/; all of its state lives under ~/.gt/.
  • gt does not modify your working tree; edits go to the overlay directory instead.
  • After a git pull or git rebase, a view's pinned branch may have moved; a gt umount followed by gt mount re-indexes to the new tree.
  • Push and fetch to remotes are entirely the job of plain git — gt is offline-only.
