ClearCase feel, Git inside.

gt wraps a standard Git repository with the workflow a cleartool user expects — named views, reserved and unreserved checkouts, dynamic mounts that keep each view's edits isolated. No central VOB server, no kernel extension, no changes to your repo.

Install & start in 5 minutes See real workflows

Why gt

Independent views

Each view is a persistent workspace pinned to a branch. Two views on the same branch keep fully separate in-progress edits, mtimes, and checkout claims.

Real checkout semantics

Reserved checkouts give you an exclusive edit lock across views; unreserved ones allow parallel work with first-to-ci wins. Git doesn't offer this on its own.

Lightweight mounts

A view's tree is served through a user-space virtual file system. Files are read lazily — no whole-repo materialization, no wasted disk space per view.

Per-view mtime preservation

Every view maintains file modification times independently. Switching views does not disturb any other view's state.

Stays out of your repo

gt's metadata lives in ~/.gt/ — never in .git/. Your normal git tooling and IDE integrations keep working unchanged.

Real Git commits

When you check in, gt creates an ordinary Git commit on the view's pinned branch. git log, git push, and CI systems see it as any other commit.

At a glance

# one-time setup per repo
gt mkvob

# a view pinned to main
gt mkview dev --branch main
gt mount  dev

# in another terminal — claim a file, edit it, check in
gt co  src/hello.c --view dev -c "adding hello"
$EDITOR ~/.gt/repos/<id>/mounts/dev/src/hello.c
gt ci  src/hello.c --view dev -c "implemented hello"

# push the commit as you would any git commit
git -C ~/src/myproject push origin main
