Using gt on an existing Git repository

A scenario-driven walkthrough. Each scenario shows the plain-git way and the gt way side by side, so you can see exactly what changes and what stays the same.

Mental model in 5 minutes

gt layers a workspace plane on top of an ordinary Git repository. The repository itself is never modified by gt.

gt conceptClosest git analogWhat it is
view A worktree with independent staging A named workspace with its own mount point and its own list of in-progress edits. Two views can follow the same branch independently.
config spec The branch that view looks at Which branch the view pins to. A single branch name in Phase 1.
checkout (no git analog) A per-file "I claim this" record. Reserved mode is an exclusive lock; unreserved allows parallel claims.
overlay Staging area + working tree The per-view area where checked-out file edits live. Visible only through that view's mount.
ci git add + git commit Converts overlay contents to a real Git commit and advances the view's branch tip.
The one-liner: gt ci produces an ordinary Git commit. git log, git push, and git blame treat it identically to any other commit.

Scenarios

S1 Attaching gt to a repo you already use

You have an existing Git project and want to try gt on it without disturbing your working tree, branches, or remote.

Do

cd ~/src/myproject
gt mkvob
gt mkview dev --branch main
gt mount dev                              # blocks until umount

Result The repo tree is served read-only at ~/.gt/repos/<id>/mounts/dev/. Your original working tree is untouched; git status is clean.

S2 Try a risky change without polluting the working tree

You want to experiment on main but your working tree has other in-flight edits. The usual answer is stash → branch → edit → revert.

Plain git

git stash
git checkout -b experiment main
vim src/hello.c
# decide to bail
git checkout main
git branch -D experiment
git stash pop

gt — working tree stays put

gt mkview exp --branch main
gt mount exp
# other terminal:
gt co src/hello.c --view exp -c "try"
$EDITOR <mount>/src/hello.c
# decide to bail
gt unco src/hello.c --view exp
gt umount exp
gt rmview exp

Result Your original working tree was never modified. git status is exactly as before.

S3 Two parallel pieces of work

You're building feature A and feature B at the same time and need to hop between them.

Plain git — branch juggling

git checkout -b feat-a main
vim src/a.c
git stash

git checkout -b feat-b main
vim src/b.c
git stash

git checkout feat-a
git stash pop
# ... repeat

gt — keep both views alive

# one-time setup
gt mkview feat-a --branch feat-a
gt mkview feat-b --branch feat-b
gt mount feat-a &
gt mount feat-b &

# work in whichever mount you like; no stash
cd <feat-a-mount>     # on A
cd <feat-b-mount>     # on B

Result Each view keeps its own edits, mtimes, and checkout list. Switching is free.

S4 "I'm editing this file — don't touch"

You want to explicitly prevent any other view from editing a critical file while you work on it.

Do

gt co src/critical.conf --view main -reserved -c "urgent fix"

# Another view attempting the same is rejected immediately:
gt co src/critical.conf --view other
#   → checkout conflict: element 'src/critical.conf' is already
#     checked out (reserved requires no other holders)

Result Other views on the same machine cannot start editing. Enforced at the storage layer with a uniqueness constraint.

S5 Multiple views loosely editing the same file

You'd rather not lock. Whoever commits first wins; others reconcile.

Do

gt co README.md --view a -unreserved -c "intro rewrite"
gt co README.md --view b -unreserved -c "add FAQ"

# view a commits first
gt ci README.md --view a

# view b sees the branch has moved
gt ci README.md --view b
#   → warning: branch 'main' moved since checkout (unreserved).
#     Committing on top of current tip.

S6 Peek at an old version without losing current edits

You're mid-edit on main but want to browse a release branch from a month ago.

Do

gt mkview peek --branch release-2025-10
gt mount peek
# when done
gt umount peek
gt rmview peek

Result Your working view is completely untouched. No git checkout / git switch required to inspect an old snapshot.

S7 Retarget a view to a different branch

Move an existing view from develop to release-2026-04.

Do

gt ls --co --view dev           # inspect outstanding checkouts
gt ci foo.c --view dev          # (or gt unco)

gt setcs dev --branch release-2026-04
gt umount dev
gt mount dev

S8 Pulling upstream changes

A teammate pushed commits to main; you want to pick them up without losing your in-progress checkouts.

Do

# gt doesn't do network ops. Use plain git.
cd ~/src/myproject
git pull origin main

# The FUSE daemon indexes the tree at mount time, so remount:
gt umount dev
gt mount dev
# Your overlays are preserved — in-progress edits survive.

S9 Inspect history and diffs

See when and why a file changed; compare your in-progress edits to the branch tip.

Do

gt lsh src/hello.c --view dev -n 10     # gt-scoped log
gt diff src/hello.c --view dev          # overlay vs branch tip

# Anything gt doesn't wrap, use plain git:
git -C ~/src/myproject log -- src/hello.c
git -C ~/src/myproject blame src/hello.c

S10 Abandoning an edit

You decided not to ship a change.

Do

gt unco src/hello.c --view dev              # silent discard
gt unco src/hello.c --view dev -keep        # keep a .keep backup
gt rmview dev -f                            # nuke the whole view

S11 Push your gt commits upstream

gt ci moved the local branch tip. You want it on GitHub / your internal remote.

Do

cd ~/src/myproject
git log --oneline main -5            # see the commits gt created
git push origin main

Result Your gt commits are on the remote. Teammates see them as ordinary git commits — because they are.

git ↔ gt cheat sheet

GoalgitgtNotes
Create workspace git worktree add ../feat feat gt mkview feat --branch feat Uses a virtual mount, not a disk copy.
Switch workspace git checkout main (unnecessary — views persist) No stash required.
Start editing (git lets you just edit) gt co <file> --view V Explicit checkout as a safety gate.
Commit git add X; git commit gt ci X --view V -c "msg" gt ci produces a normal git commit.
Discard edit git restore X gt unco X --view V -keep preserves a backup.
See status git status gt ls --co --view V Shows the view's checkouts.
History git log -- X gt lsh X --view V Scoped to the view's branch.
Diff git diff X gt diff X --view V Overlay vs branch tip.
Switch branch git switch other gt setcs V --branch other + remount Remount applies the change.
Pull upstream git pull git pull + gt umount/mount gt is offline-only.
Push git push git push (unchanged) Same as always.
Stash git stash (unnecessary — overlay is the stash) Keep the view and edits are never lost.
Blame git blame X use git blame gt does not wrap it.
Rebase / merge git rebase git rebase Then gt umount/mount to resync.

Coexisting with plain git

gt never touches .git/. You can freely mix gt and plain git commands on the same repository. A few things to keep in mind:

  • Read-only git (log, blame, show) is always safe.
  • git pull / git push work as usual — remount a view when its pinned branch has moved.
  • Deleting the branch a view points to (git branch -D) will break the next gt mount; retarget with gt setcs first.
  • Editing the same file simultaneously through the FUSE mount and your plain working tree is just confusing — pick one.

Common pitfalls

"The mount exists but I can't write to files"

The mount is read-only by default. Run gt co <file> first. This gate is intentional — it prevents accidental edits.

"I ci'd but the mount still shows the old content"

The FUSE daemon indexes the tree at mount time. Run gt umount then gt mount to pick up the new tip.

"gt mount keeps my terminal blocked"

Use a dedicated terminal window, run with the shell's & (gt mount dev &), or keep it in a tmux/screen session.

"Can I skip --view everywhere?"

Yes — every element command honors the GT_VIEW environment variable. gt setview <name> drops you into a subshell with it set.
