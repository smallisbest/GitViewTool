gt는 어떻게 만들어졌나

gt는 단일 자체 완결 커맨드라인 바이너리입니다. 저장소를 수정하지 않고, 별도 디렉터리에 자체 상태를 유지하며, 기존 Git 내용 위에 view별 가상 파일시스템을 제공합니다.

디스크 레이아웃

gt가 관리하는 모든 상태는 ~/.gt/ 아래에 있습니다 (GT_HOME 환경변수로 override 가능):

~/.gt/                          # $GT_HOME으로 override
├── registry.db                 # 저장소 경로 → repo-id 인덱스
├── log/
│   └── gt.log                  # rotating 로그 (5 × 1 MB)
└── repos/<12-hex-id>/
    ├── metadata.db             # views, checkouts
    ├── config.toml             # default_cs_branch, default_remote
    ├── views/<view>/
    │   └── overlay/            # 쓰기 가능 shadow
    └── mounts/<view>/          # 기본 FUSE 마운트 경로

gt mkvob로 등록된 각 저장소는 정규 절대 경로의 12자리 16진 digest로 식별되는 자체 하위 디렉터리를 갖습니다. 저장소 경로가 바뀌면 gt 입장에서는 다른 저장소로 인식됩니다.

View 격리

View는 다음 세 가지 상태로 구성됩니다:

  1. 메타데이터 DB의 레코드: 이름, 종류(Phase 1은 모두 dynamic), 고정 브랜치, 마운트 경로.
  2. overlay — 체크아웃된 파일의 편집 내용이 저장되는 실제 디렉터리.
  3. 마운트 경로 — FUSE 백엔드가 view를 노출하는 디렉터리.

같은 브랜치 위의 두 view는 완전히 독립적입니다. 수정되지 않은 파일 내용은 Git을 통해 공유(같은 blob)하지만, overlay, 체크아웃 레지스트리, 마운트 경로는 각각 분리되어 있습니다.

FUSE 레이어

gt mount <view>를 실행하면 FUSE 백엔드가 다음 작업을 수행합니다:

  1. View의 고정 브랜치를 commit OID로 해석.
  2. 해당 commit의 트리를 순회해 마운트 내 절대 경로로 인덱싱된 인메모리 인덱스 구축.
  3. POSIX 파일시스템 호출에 응답:
    • getattr / readdir — 인메모리 인덱스에서 응답. 수정 시간은 커밋의 타임스탬프.
    • open / read — 최초 접근 시 libgit2로 blob을 지연 로드하고 프로세스 내에서 캐시.
    • write / truncate — 해당 경로에 활성 체크아웃이 있을 때만 허용. 쓰기는 overlay 파일로 감.
    • readlink — 심볼릭 링크 엔트리의 blob 내용을 반환.

macOS에서는 FUSE-T의 NFSv4 루프백 전송 계층을 통해 백엔드와 통신하므로 커널 확장이 필요하지 않습니다.

체크아웃 강제

gt의 체크아웃 레지스트리는 저장소별 SQLite DB에 있으며 두 가지 규칙을 강제합니다:

  • Reserved 배타성 — (element_path) WHERE mode = 'reserved'에 걸린 partial unique index가 특정 요소에 대해 한 번에 하나의 view만 reserved 체크아웃을 보유할 수 있도록 보장.
  • Unreserved 동시성 — 여러 view가 같은 요소에 동시에 unreserved 체크아웃을 보유 가능. 먼저 커밋한 view가 이기고, 이후 체크인은 기준 브랜치가 움직였다는 경고와 함께 tip 위에 얹힘.

FUSE 백엔드는 모든 쓰기 경로에서 이 레지스트리를 참조합니다. 체크아웃 없는 파일은 open 시점에 EROFS로 거부되어 실수 편집을 방지합니다.

체크인 동작

  1. Overlay 파일의 바이트 읽기.
  2. libgit2로 새 Git blob 작성.
  3. 현재 브랜치 tip 트리를 복제하며 해당 요소의 leaf를 새 blob OID로 교체.
  4. 사용자의 user.name / user.email Git 설정으로 author를 채운 커밋을 브랜치 tip을 부모로 하여 생성.
  5. 브랜치 참조를 새 커밋으로 fast-forward.
  6. 체크아웃 기록 제거, overlay 파일 삭제.

결과물은 일반 Git으로 만든 커밋과 구별되지 않습니다. git log, git push, CI 파이프라인 모두 동일하게 취급합니다.

일반 git과 공존

gt와 일반 git은 같은 작업 저장소를 안전하게 공유할 수 있습니다:

  • gt는 .git/을 전혀 건드리지 않고, 자체 상태는 ~/.gt/에만 존재.
  • gt는 working tree를 수정하지 않고, 편집은 overlay 디렉터리로 들어감.
  • git pull이나 git rebase 후 view의 고정 브랜치가 움직였을 수 있으므로 gt umount + gt mount로 새 트리를 다시 인덱싱.
  • 리모트로의 push/fetch는 전적으로 일반 git의 역할 — gt는 오프라인 전용.
