기존 Git 저장소에서 gt 쓰기

시나리오 중심의 실전 가이드. 각 시나리오에서 "원래 git으로는 이렇게 했는데 gt로는 이렇게 한다"를 좌우로 비교해, 무엇이 달라지고 무엇이 그대로인지 한눈에 파악할 수 있습니다.

5분 개념 모델

gt는 일반 Git 저장소 위에 작업공간 계층을 얹습니다. 저장소 자체는 건드리지 않습니다.

gt 개념git 유사 개념무엇인가
view 독립 staging을 가진 worktree 이름 붙은 작업공간. 각자의 마운트 경로와 편집 중 파일 목록을 가짐. 두 view가 같은 브랜치를 독립적으로 볼 수 있음.
config spec view가 바라보는 브랜치 view가 어느 브랜치를 기준으로 삼을지. Phase 1에서는 단일 브랜치 이름.
checkout (git에 대응 없음) 파일 단위 "이 파일 내가 편집 중" 선언. Reserved는 배타 락, unreserved는 동시 허용.
overlay staging + working tree 체크아웃된 파일의 편집 내용이 저장되는 view 전용 영역. 해당 view의 마운트에서만 보임.
ci git add + git commit overlay 내용을 진짜 Git 커밋으로 만들고 view의 브랜치 tip을 이동.
핵심 한 줄: gt ci가 만드는 커밋은 일반 Git 커밋과 동일합니다. git log, git push, git blame 모두 구분하지 않습니다.

시나리오

S1 기존 저장소에 gt 처음 붙이기

이미 쓰던 Git 프로젝트에 gt를 얹고 싶다. working tree, 브랜치, 리모트 모두 건드리지 않고.

하기

cd ~/src/myproject
gt mkvob
gt mkview dev --branch main
gt mount dev                              # umount 전까지 블록

결과 저장소 트리가 ~/.gt/repos/<id>/mounts/dev/에 읽기 전용으로 노출. 원 working tree는 건드리지 않음. git status는 깨끗.

S2 working tree를 오염시키지 않고 실험적 수정

main에서 뭔가 고쳐보고 싶은데 현재 working tree에는 다른 편집이 쌓여 있다. 원래라면 stash → 새 브랜치 → 수정 → 되돌리기 해야 한다.

원래 git 방식

git stash
git checkout -b experiment main
vim src/hello.c
# 버리기로 결정
git checkout main
git branch -D experiment
git stash pop

gt — working tree 그대로

gt mkview exp --branch main
gt mount exp
# 다른 터미널:
gt co src/hello.c --view exp -c "try"
$EDITOR <mount>/src/hello.c
# 버리기로 결정
gt unco src/hello.c --view exp
gt umount exp
gt rmview exp

결과 원 working tree는 전혀 수정 안 됨. git status가 처음과 동일.

S3 두 가지 작업을 병렬로 진행

기능 A와 B를 동시에 만들고 있고 자주 오가야 한다.

원래 git — 브랜치 왕복

git checkout -b feat-a main
vim src/a.c
git stash

git checkout -b feat-b main
vim src/b.c
git stash

git checkout feat-a
git stash pop
# ... 반복

gt — view 둘 다 살려둠

# 셋업은 한 번만
gt mkview feat-a --branch feat-a
gt mkview feat-b --branch feat-b
gt mount feat-a &
gt mount feat-b &

# 원하는 마운트로 cd, stash 필요 없음
cd <feat-a-mount>     # A 작업
cd <feat-b-mount>     # B 작업

결과 각 view의 편집 상태, mtime, 체크아웃 목록이 독립 유지. 스위칭 비용 0.

S4 "이 파일 내가 수정 중" 명시적 락

중요한 설정 파일을 다른 view가 실수로 동시에 수정하는 것을 막고 싶다.

하기

gt co src/critical.conf --view main -reserved -c "urgent fix"

# 다른 view의 시도는 즉시 거부:
gt co src/critical.conf --view other
#   → checkout conflict: element 'src/critical.conf' is already
#     checked out (reserved requires no other holders)

결과 같은 머신의 다른 view가 수정 착수 불가. 메타데이터 레벨의 unique 제약으로 보장.

S5 여러 view가 같은 파일을 느슨하게 편집

락을 걸고 싶지 않음. 먼저 커밋한 쪽이 이기고 나머지는 재조정.

하기

gt co README.md --view a -unreserved -c "intro rewrite"
gt co README.md --view b -unreserved -c "add FAQ"

# view a가 먼저 체크인
gt ci README.md --view a

# view b는 브랜치가 움직였다는 경고를 봄
gt ci README.md --view b
#   → warning: branch 'main' moved since checkout (unreserved).
#     Committing on top of current tip.

S6 과거 버전을 잠깐 열어보기

지금 main에서 작업 중인데 한 달 전 release 브랜치 상태를 보고 싶다.

하기

gt mkview peek --branch release-2025-10
gt mount peek
# 확인 끝
gt umount peek
gt rmview peek

결과 현재 작업 view는 영향 없음. git checkout/git switch 없이 과거 열람.

S7 view의 기준 브랜치 변경

기존 view를 develop에서 release-2026-04로 돌리고 싶다.

하기

gt ls --co --view dev           # 체크아웃 상태 확인
gt ci foo.c --view dev          # (또는 gt unco)

gt setcs dev --branch release-2026-04
gt umount dev
gt mount dev

S8 리모트에서 받아오기

팀원이 main에 커밋을 올렸다. 현재 체크아웃을 유지하면서 받아오고 싶다.

하기

# gt는 네트워크 조작 안 함. 일반 git 사용.
cd ~/src/myproject
git pull origin main

# FUSE 데몬은 마운트 시점 트리를 사용하므로 재마운트:
gt umount dev
gt mount dev
# overlay는 유지됨 — 편집 내용 안 사라짐.

S9 이력과 diff 확인

파일이 언제 왜 바뀌었는지 보고, 내 편집이 브랜치 tip과 얼마나 다른지 확인.

하기

gt lsh src/hello.c --view dev -n 10     # view 기준 간결 로그
gt diff src/hello.c --view dev          # overlay vs 브랜치 tip

# gt가 안 감싸는 기능은 일반 git으로:
git -C ~/src/myproject log -- src/hello.c
git -C ~/src/myproject blame src/hello.c

S10 편집 버리기

실험한 편집을 날리고 싶다.

하기

gt unco src/hello.c --view dev              # 조용히 버림
gt unco src/hello.c --view dev -keep        # .keep 백업 남김
gt rmview dev -f                            # view 통째로 폐기

S11 만든 커밋을 리모트에 push

gt ci로 로컬 브랜치 tip이 움직였다. 리모트에 올리고 싶다.

하기

cd ~/src/myproject
git log --oneline main -5            # gt가 만든 커밋 확인
git push origin main

결과 리모트에 올라감. 팀원 입장에서는 평범한 Git 커밋.

git ↔ gt 대응표

목적gitgt주의
작업공간 생성 git worktree add ../feat feat gt mkview feat --branch feat 디스크 사본 대신 가상 마운트.
작업공간 전환 git checkout main (불필요 — view 유지됨) stash 필요 없음.
편집 시작 (자유롭게 편집) gt co <file> --view V 명시적 체크아웃 = 실수 방지 gate.
커밋 git add X; git commit gt ci X --view V -c "msg" 일반 Git 커밋 생성.
편집 취소 git restore X gt unco X --view V -keep로 백업 가능.
상태 보기 git status gt ls --co --view V view의 체크아웃만 표시.
이력 git log -- X gt lsh X --view V view 기준 브랜치만.
diff git diff X gt diff X --view V overlay vs 브랜치 tip.
브랜치 전환 git switch other gt setcs V --branch other + 재마운트 재마운트로 반영.
리모트 업데이트 git pull git pull + gt umount/mount gt는 오프라인 전용.
push git push git push (그대로) 동일.
stash git stash (불필요 — overlay가 자연스러운 stash) view 유지만 해도 편집 안 사라짐.
blame git blame X git blame 그대로 gt는 감싸지 않음.
rebase / merge git rebase git rebase 이후 gt umount/mount로 동기화.

git과 공존

gt는 .git/을 전혀 건드리지 않습니다. 같은 저장소에서 gt와 일반 git 명령을 섞어 써도 문제 없지만 몇 가지는 알아두면 좋습니다.

  • 읽기 전용 git (log, blame, show)은 언제든 안전.
  • git pull, git push는 그대로 사용 — view의 브랜치가 움직이면 재마운트하여 반영.
  • View가 가리키는 브랜치를 git branch -D로 삭제하면 다음 gt mount가 실패 — 먼저 gt setcs로 다른 브랜치 지정.
  • 같은 파일을 FUSE 마운트와 원래 working tree에서 동시에 편집하면 혼란만 생김. 한 쪽에서만.

자주 하는 실수

"마운트는 됐는데 쓰기가 안 돼요"

마운트는 기본 읽기 전용. 먼저 gt co <file> 해야 합니다. 의도된 동작 — 실수 편집 방지.

"ci 했는데 마운트에서 여전히 옛날 내용이 보여요"

FUSE 데몬이 마운트 시점의 트리를 사용합니다. gt umount 후 gt mount로 새 tip 반영.

"터미널이 gt mount로 블록돼요"

별도 터미널 창, 셸의 & (gt mount dev &), 또는 tmux/screen 세션 사용.

--view 생략하고 싶어요"

모든 파일 명령이 GT_VIEW 환경변수를 인식합니다. gt setview <name>은 이 변수를 세팅한 서브셸을 열어줍니다.
